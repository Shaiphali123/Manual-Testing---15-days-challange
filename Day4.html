üöÄ What is Agile?

Agile is a software development methodology that emphasizes iterative development, team collaboration, customer feedback, and responding to change. Unlike traditional waterfall methods, Agile delivers working software in short iterations called sprints.

üí° Agile Manifesto Values

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

üìã Agile Principles (12 Key Principles)

  1. Our highest priority is to satisfy the customer through early and continuous delivery
  2. Welcome changing requirements, even late in development
  3. Deliver working software frequently (weeks rather than months)
  4. Business people and developers must work together daily
  5. Build projects around motivated individuals and trust them
  6. Face-to-face conversation is the most efficient communication
  7. Working software is the primary measure of progress
  8. Sustainable development pace for teams
  9. Continuous attention to technical excellence and good design
  10. Simplicity - maximizing the amount of work not done
  11. Self-organizing teams produce the best results
  12. Regular team retrospectives to improve effectiveness

üèÉ‚Äç‚ôÇÔ∏è Scrum Framework

üèÉ‚Äç‚ôÇÔ∏è Scrum - Most Popular Agile Framework

Overview: Scrum is an iterative and incremental framework for managing product development. It structures development in cycles called Sprints, typically 1-4 weeks long.

üë• Scrum Roles:

üéØ Product Owner
  • Defines product vision and roadmap
  • Manages product backlog
  • Prioritizes features based on business value
  • Acts as customer representative
  • Accepts or rejects completed work
üèÉ‚Äç‚ôÇÔ∏è Scrum Master
  • Facilitates Scrum process
  • Removes impediments for the team
  • Coaches team on Agile practices
  • Ensures Scrum rules are followed
  • Protects team from external interruptions
üë®‚Äçüíª Development Team
  • Cross-functional team (developers, testers, designers)
  • Self-organizing and self-managing
  • Delivers potentially shippable product increment
  • Estimates effort for user stories
  • Commits to sprint goals

üìÖ Scrum Events (Ceremonies):

üóìÔ∏è Sprint Planning (4-8 hours for 2-week sprint)

  • Purpose: Plan work for the upcoming sprint
  • Participants: Entire Scrum team
  • Output: Sprint backlog and sprint goal
  • Testing Input: Testers help estimate testing effort and identify testability concerns

üåÖ Daily Standup (15 minutes)

  • Purpose: Synchronize team activities and plan for next 24 hours
  • Three Questions: What did you do yesterday? What will you do today? Any impediments?
  • Testing Focus: Testers share progress on testing tasks, blockers, and help needed

üìä Sprint Review (2-4 hours)

  • Purpose: Demonstrate completed work to stakeholders
  • Focus: Show working software increment
  • Testing Role: Testers present test results and quality metrics

üîÑ Sprint Retrospective (1-3 hours)

  • Purpose: Reflect on process and identify improvements
  • Format: What went well? What could be improved? Action items?
  • Testing Improvements: Discuss testing processes, tools, and team collaboration

üìã Scrum Artifacts:

  • Product Backlog: Prioritized list of features/requirements
  • Sprint Backlog: Items selected for current sprint + plan to deliver them
  • Product Increment: Working, tested software at end of sprint

üìñ User Stories & Acceptance Criteria

üìù User Story Format

"As a [type of user], I want [some goal] so that [some reason/benefit]"

üåü Example User Story:

Story: As a registered user, I want to reset my password so that I can regain access to my account if I forget it.

‚úÖ Acceptance Criteria:

  • Given I'm on the login page, when I click "Forgot Password", then I should see a password reset form
  • Given I enter a valid email, when I submit the form, then I should receive a password reset email
  • Given I click the reset link in email, when I enter a new password, then my password should be updated
  • Given I enter an invalid email, when I submit the form, then I should see an error message

üß™ Testing in Agile

üß™ Agile Testing Principles

üéØ Key Testing Principles in Agile:

  • Continuous Testing: Testing throughout the sprint, not just at the end
  • Shift Left: Start testing activities early in the development cycle
  • Whole Team Approach: Everyone is responsible for quality
  • Fast Feedback: Quick feedback loops between dev and test
  • Risk-Based Testing: Focus testing on high-risk areas
  • Automated Testing: Heavy emphasis on test automation

üîç Definition of Done (DoD)

‚úÖ Sample Definition of Done

A user story is considered "Done" when:

  • ‚úÖ Code is written and reviewed
  • ‚úÖ Unit tests are written and passing
  • ‚úÖ Integration tests are passing
  • ‚úÖ Functional testing is completed
  • ‚úÖ Acceptance criteria are met
  • ‚úÖ No critical or high severity bugs
  • ‚úÖ Code is deployed to staging environment
  • ‚úÖ Documentation is updated
  • ‚úÖ Product Owner has accepted the story

üîß Agile Testing Techniques

üéØ Test-Driven Development (TDD)

  1. Red: Write a failing test case
  2. Green: Write minimum code to pass the test
  3. Refactor: Improve code while keeping tests green

Benefits: Better code quality, comprehensive test coverage, fewer defects

üîÑ Behavior-Driven Development (BDD)

Format: Given-When-Then scenarios

  • Given: Initial context/preconditions
  • When: Action performed
  • Then: Expected outcome

Example: Given user is logged in, When user clicks logout, Then user should be redirected to login page

üöÄ Exploratory Testing

  • Simultaneous learning, test design, and execution
  • Unscripted testing approach
  • Focus on discovery and investigation
  • Time-boxed sessions (30-90 minutes)
  • Document findings and charter

üìà Agile Testing Metrics

üìä Key Metrics for Agile Teams

  • Velocity: Story points completed per sprint
  • Burndown Charts: Visual representation of work remaining
  • Defect Escape Rate: Bugs found in production vs total bugs
  • Test Coverage: Percentage of code covered by tests
  • Cycle Time: Time from story start to completion
  • Customer Satisfaction: Feedback from end users

‚ö†Ô∏è Common Agile Testing Challenges

üö® Challenges & Solutions

  • Challenge: Insufficient time for testing
    Solution: Start testing early, automate repetitive tests, focus on risk-based testing
  • Challenge: Changing requirements
    Solution: Embrace change, maintain flexible test cases, continuous communication
  • Challenge: Lack of documentation
    Solution: Living documentation, automated tests as documentation, collaborative sessions
  • Challenge: Test environment issues
    Solution: Infrastructure as code, containerization, environment automation
  • Challenge: Team coordination
    Solution: Daily standups, clear communication channels, shared responsibilities

üéØ Best Practices for Agile Testing

‚úÖ Agile Testing Best Practices

  • Start Testing Early: Begin testing activities from sprint planning
  • Automate Regression: Automate repetitive and regression tests
  • Risk-Based Testing: Focus on high-risk areas first
  • Continuous Integration: Integrate code and run tests frequently
  • Collaborate Actively: Work closely with developers and product owners
  • Test in Production-like Environment: Use environments similar to production
  • Maintain Test Data: Keep test data fresh and relevant
  • Learn Continuously: Adapt and improve testing practices
