üìö What is Software Testing?

Software testing is a systematic process of evaluating, verifying, and validating a software application to ensure it meets specified requirements and functions correctly. It involves executing software components using manual or automated tools to evaluate one or more properties of interest.

Testing is not just about finding bugs - it's about building confidence in the software quality. It helps determine whether the actual results match the expected results and ensures the software system is defect-free.

üí° Formal Definition (ISTQB)

Software Testing is a process consisting of all lifecycle activities, both static and dynamic, concerned with planning, preparation and evaluation of software products and related work products to determine that they satisfy specified requirements, to demonstrate that they are fit for purpose and to detect defects.

üéØ Goals of Software Testing

Primary Goals:

  • Verification: Are we building the product right?
  • Validation: Are we building the right product?
  • Error Detection: Find defects before users do
  • Risk Mitigation: Reduce the risk of failures in production
  • Quality Assessment: Provide information about software quality

üé≠ Seven Principles of Software Testing

Principle 1: Testing shows the presence of defects

Testing can prove that defects are present, but cannot prove that there are no defects. Testing reduces the probability of undiscovered defects remaining in the software but, even if no defects are found, testing is not a proof of correctness.

Principle 2: Exhaustive testing is impossible

Testing everything (all combinations of inputs and preconditions) is not feasible except for trivial cases. Rather than attempting to test exhaustively, risk analysis, test techniques, and priorities should be used to focus test efforts.

Principle 3: Early testing saves time and money

To find defects early, both static and dynamic test activities should be started as early as possible in the software development lifecycle. Early testing is sometimes referred to as "shift left."

Principle 4: Defects cluster together

A small number of modules usually contain most of the defects discovered during pre-release testing, or are responsible for most of the operational failures (Pareto Principle - 80% of problems come from 20% of modules).

Principle 5: Beware of the pesticide paradox

If the same tests are repeated over and over again, eventually these tests no longer find new defects. To detect new defects, existing tests and test data may need to be changed, and new tests may need to be written.

Principle 6: Testing is context dependent

Testing is done differently in different contexts. For example, safety-critical industrial control software is tested differently from an e-commerce mobile app. Testing approaches and techniques should be appropriate for the context.

Principle 7: Absence-of-errors is a fallacy

Finding and fixing defects does not help if the system built is unusable and does not fulfill the user's needs and expectations. Verification and validation should be performed to ensure the system meets user requirements.

üîÑ Static vs Dynamic Testing

Static Testing

  • Testing without executing code
  • Examples: Code reviews, walkthroughs, inspections
  • Tools: SonarQube, Checkmarx, Veracode
  • Finds: Logic errors, syntax errors, security vulnerabilities

Dynamic Testing

  • Testing by executing code
  • Examples: Functional testing, performance testing
  • Tools: Selenium, JMeter, Postman
  • Finds: Runtime errors, performance issues, integration problems

üèóÔ∏è Testing Pyramid

The testing pyramid illustrates the ideal distribution of different types of tests:

  • Unit Tests (Base): 70% - Fast, cheap, numerous
  • Integration Tests (Middle): 20% - Medium speed and cost
  • UI/E2E Tests (Top): 10% - Slow, expensive, fewer

üë®‚Äçüíª Types of Testers

üéØ Manual Testers

  • Execute test cases manually without automation tools
  • Focus on user experience and exploratory testing
  • Excel at finding usability issues and edge cases
  • Essential for creative and intuitive testing scenarios

ü§ñ Automation Testers

  • Create and maintain automated test scripts
  • Focus on regression testing and repetitive tasks
  • Use tools like Selenium, Cypress, TestComplete
  • Complement manual testing efforts

üëë Role of a Manual Tester

Daily Responsibilities:

  • Requirements Analysis: Study and understand business requirements
  • Test Planning: Create test strategies and test plans
  • Test Case Design: Write detailed, executable test cases
  • Test Execution: Execute test cases and document results
  • Defect Management: Report, track, and verify bug fixes
  • Communication: Collaborate with developers, BAs, and stakeholders
  • Documentation: Maintain test artifacts and reports
  • Continuous Learning: Stay updated with testing trends and tools

üìù Today's Comprehensive Tasks

‚úèÔ∏è Task 1: Testing Mindset Development

Choose a mobile app you use daily (WhatsApp, Instagram, Banking app, etc.):

  1. Positive Testing: List 5 features that work perfectly
  2. Negative Scenarios: Try 3 invalid inputs and note behavior
  3. Usability Issues: Identify 3 areas that could be improved
  4. Security Concerns: List 2 potential security risks
  5. Performance: Note any slow-loading features

‚úèÔ∏è Task 2: Real-World Bug Hunting

Visit any e-commerce website (Amazon, Flipkart, etc.) and test the search functionality:

  • Search with valid product names
  • Search with invalid/gibberish text
  • Search with special characters (@#$%)
  • Leave search box empty and click search
  • Try very long search terms (100+ characters)

Document: What happens in each case? Any unexpected behavior?

‚úèÔ∏è Task 3: Professional Documentation

Create your first simple test case using this format:

  • Test Case ID: TC_001
  • Test Scenario: User Login Functionality
  • Pre-conditions: User should be registered
  • Test Steps:
    1. Navigate to login page
    2. Enter valid email
    3. Enter valid password
    4. Click Login button
  • Expected Result: User should be successfully logged in
  • Actual Result: [To be filled during execution]
  • Status: [Pass/Fail]
